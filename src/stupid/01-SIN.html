Singleton

Si crees que necesitas un Singleton...

probablemente no lo necesites.

Por qué?

  • Expone estado global (global is evil!)
  • Esconde dependencias
  • Dificulta Unit Testing
  • Crea acoplamiento

Veámoslo en un ejemplo...


class DBLogger
{
  /** @var PSQLConnection */
  private $connection;

  /** @var MyDbConnection */
  private $instance = null;

  private function __construct() {
    $this->connection = new PSQLConnection();
  }

  public static function getInstance() {
    if (instance === null) {
      $this->instance = new static;
    }
    return instance;
  }

  public function log($message) {
    // insert message in log db
  }
}

¿Por qué esta mal?

  • Rompe la arquitectura del proyecto
  • Si se modifica en algun punto, su estado cambia para el resto de la App
  • Hace imposible testear clases que usen este singleton sin hacer cambios en la db

¿Que dependencias esconde?

  • private $connection (PSQLConnection)
    connection esta "atada" al tipo específico de DB
  • Esto rompe con los principios SRP y O/C...que veremos mas adelante

Como impacta en Unit Testing?

Supongamos la siguiente secuencia:

  1. Corremos un test que use la Db
  2. Da VERDE
  3. Algun proceso modifica nuestro objeto MyDbConnection
  4. Corremos el mismo test
  5. Puede dar ROJO
